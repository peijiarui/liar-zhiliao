# 两级缓存（Two-Tier Cache）工作流程

## 一、涉及的类和模块

| 类 | 所属模块 | 角色 |
|------|----------|------|
| `CacheConfig` | `zhiliao-app` | **组装车间**：声明所有缓存相关 Bean |
| `TwoTierCacheManager` | `zhiliao-app` | **代理经理**：实现 `CacheManager`，对外提供两级缓存 |
| `TwoTierCache` | `zhiliao-app` | **实际执行者**：实现 `Cache`，内部持有 L1 Caffeine + L2 Redis |
| `RetrievalCacheService` | `zhiliao-retrieval` | **调用方**：标注 `@Cacheable`/`@CachePut`/`@CacheEvict`，声明式使用缓存 |

## 二、类的关联方式（Spring Cache 抽象）

这几个类表面上没有直接引用关系，连接的"胶水"是 **Spring 缓存 AOP 机制**：

```
@EnableCaching                    ← 在 CacheConfig 上，开启 Spring 缓存能力
       ↓
Spring 扫描 @Cacheable/@CachePut  ← 在 RetrievalCacheService 上
       ↓
为 RetrievalCacheService 创建 AOP 动态代理
       ↓
AOP 拦截器需要找一个 CacheManager
       ↓
找到 @Primary 的 twoTierCacheManager  ← TwoTierCacheManager（声明在 CacheConfig 中）
       ↓
按缓存名 "qa_answer" 获取 Cache 实例    ← TwoTierCache（由 TwoTierCacheManager 内部创建）
       ↓
调用 Cache.get(key) / Cache.put(key, value)
       ↓
内部路由到 L1 CaffeineCache 或 L2 RedisCache
```

**核心思想：** `RetrievalCacheService` 只需要写一行 `@Cacheable("qa_answer")`，Spring 框架在运行时自动串联起 `TwoTierCacheManager` → `TwoTierCache` → L1/L2 的全链路。这就是**声明式缓存**——只声明规则，框架负责路由。

## 三、读缓存流程：getCachedAnswer("请假流程")

```
RetrievalCacheService.getCachedAnswer("请假流程")
  │  @Cacheable(value = "qa_answer", key = "MD5(请假流程)")
  │
  ▼
[Spring CacheInterceptor（AOP 代理拦截）]
  │  计算 key = MD5("请假流程") = "abc123"
  │  找到 @Primary CacheManager → TwoTierCacheManager
  │  获取缓存区 "qa_answer" → TwoTierCacheManager.getCache("qa_answer")
  │                               │
  │                               ├─ l1.getCache("qa_answer") → CaffeineCache
  │                               └─ l2.getCache("qa_answer") → RedisCache
  │                               └─ 返回 new TwoTierCache("qa_answer", Caffeine, Redis)
  │
  ▼
TwoTierCache.get("abc123")                        ← TwoTierCache.java:38
  │
  ├─ 1️⃣ l1.get("abc123")   → Caffeine 查找（JVM 堆内，~0.01ms）
  │   └─ 命中 → 直接返回 ✅
  │
  └─ 2️⃣ l2.get("abc123")   → Redis GET cache:qa:abc123（分布式，~1ms）
      └─ 命中 → l1.put("abc123", value) ← 回填 L1（热数据提拔）
              → 返回 ✅
              │
              └─ 未命中 → 执行方法体 → getCachedAnswer 返回 null
                        → Spring 发现 null，不写入缓存
                        → 返回 null ❌
```

### key 的生成

`@Cacheable` 的 key 通过 SpEL 表达式计算，而不是调用类内的 `md5()` 方法：

```java
key = "T(org.springframework.util.DigestUtils).md5DigestAsHex(
           #question.getBytes(T(java.nio.charset.StandardCharsets).UTF_8))"
```

因此缓存 key 是 **MD5(完整用户入参)**，两次入参字符串必须完全一致才命中同一个缓存。

## 四、写缓存流程：putAnswer("请假流程", "请假需要...")

```
RetrievalCacheService.putAnswer("请假流程", "请假需要...")
  │  @CachePut(value = "qa_answer", key = "MD5(请假流程)")
  │
  ▼
[Spring CacheInterceptor（AOP 代理拦截）]
  │  计算 key = MD5("请假流程") = "abc123"
  │  获取 TwoTierCache("qa_answer")
  │  调用 cache.put("abc123", "请假需要...")
  │
  ▼
TwoTierCache.put("abc123", "请假需要...")          ← TwoTierCache.java:91
  │
  ├─ 1️⃣ l1.put("abc123", "请假需要...")  → Caffeine 写入（堆内，10min TTL）
  │
  └─ 2️⃣ l2.put("abc123", "请假需要...")  → Redis SET cache:qa:abc123（1h TTL）
```

`@CachePut` 与 `@Cacheable` 的区别：
- **`@Cacheable`**：先查缓存，命中直接返回（不执行方法体）；未命中执行方法体，返回值写入缓存
- **`@CachePut`**：总是执行方法体，然后把返回值写入缓存（用于主动更新缓存）

## 五、淘汰缓存流程：evictAnswer("请假流程")

```
TwoTierCache.evict("abc123")                      ← TwoTierCache.java:99
  ├─ l1.evict("abc123")  → Caffeine 删除
  └─ l2.evict("abc123")  → Redis DEL cache:qa:abc123
```

同时从 L1 和 L2 中删除，确保数据一致性。

## 六、整体架构图

```
┌─────────────────────────────────────────────────────────┐
│                    CacheConfig                          │
│  @Configuration + @EnableCaching                        │
│                                                         │
│  @Bean caffeineCacheManager()   ← CaffeineCacheManager  │
│  @Bean redisCacheManager()      ← RedisCacheManager     │
│  @Bean @Primary twoTierCacheManager(l1, l2)             │
│       │                                                 │
│  ┌────┴────────────────────────┐                       │
│  │ l1: CaffeineCacheManager    │  l2: RedisCacheManager │
│  └─────────┬───────────────────┘──────┬──────────────── │
└────────────┼──────────────────────────┼─────────────────┘
             │                          │
             ▼                          ▼
┌────────────────────────────────────────────────────────┐
│              TwoTierCacheManager                       │
│  getCache(name)                                        │
│    └─ caches.computeIfAbsent(name, n ->                │
│         new TwoTierCache(n,                            │
│           l1.getCache(n),   ← CaffeineCache            │
│           l2.getCache(n))   ← RedisCache               │
│                                                         │
│  L1 配置: maximumSize=1000, expireAfterWrite=10min     │
│  L2 配置: qa_answer TTL=1h, retrieval_result TTL=24h  │
└──────────────────────┬─────────────────────────────────┘
                       │
                       ▼
┌────────────────────────────────────────────────────────┐
│                  TwoTierCache                          │
│                                                        │
│  get(key)                                              │
│    ├─ l1.get(key)  ── 命中 → 返回                      │
│    └─ 未命中 → l2.get(key)                             │
│                   ├─ 命中 → l1.put(key,v) ← 回填       │
│                   │        → 返回                      │
│                   └─ 未命中 → Callable 执行 / 返回 null │
│                                                        │
│  put(key, value)                                       │
│    ├─ l1.put(key, value)                               │
│    └─ l2.put(key, value)                               │
│                                                        │
│  evict(key)                                            │
│    ├─ l1.evict(key)                                    │
│    └─ l2.evict(key)                                    │
└──────────────────────┬─────────────────────────────────┘
                       ▲
                       │ Spring AOP 代理自动路由（隐藏胶水）
                       │
┌────────────────────────────────────────────────────────┐
│              RetrievalCacheService                     │
│  (zhiliao-retrieval 模块)                               │
│                                                        │
│  Level 1 — 热点问答缓存 (qa_answer)                     │
│  ┌─────────────────────────────────────────────────┐   │
│  │ @Cacheable("qa_answer")  getCachedAnswer(q)     │   │
│  │ @CachePut("qa_answer")   putAnswer(q, a)        │   │
│  │ @CacheEvict("qa_answer") evictAnswer(q)         │   │
│  └─────────────────────────────────────────────────┘   │
│                                                        │
│  Level 2 — 检索结果缓存 (retrieval_result)              │
│  ┌─────────────────────────────────────────────────┐   │
│  │ @Cacheable("retrieval_result") getCachedRetrieval│   │
│  │ @CachePut("retrieval_result") putRetrieval(q, r) │   │
│  └─────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────┘
```

## 七、缓存配置参数

| 缓存区 | L1 Caffeine | L2 Redis | 用途 |
|--------|-------------|----------|------|
| `qa_answer` | 最大 1000 条，10min TTL | 1h TTL，Redis key 前缀 `cache:qa:` | 热点问答全文缓存 |
| `retrieval_result` | 最大 1000 条，10min TTL | 24h TTL，Redis key 前缀 `cache:retrieval:` | 检索结果（RankedChunk 列表） |

## 八、当前缓存的使用现状

### 正常流程（LLM 健康时）

```
用户提问 → ChatController.chat()
           → assistant.chat() → LLM 回答
           → LLM 决定是否调 KnowledgeRetrievalTool.retrieveKnowledge()
             → L2 缓存检索结果（命中省检索，不省 LLM）
           → LLM 组织返回
           → ❌ putAnswer 未调用，L1 热点问答缓存永不写入
```

### 熔断降级流程（LLM 异常时）

```
用户提问 → ChatController.chat()
           → @SentinelResource fallback → chatFallback()
             → L1 热点问答缓存查询（永远空，因从未写入）
             → 知识库检索返回文档片段
```

### 当前问题

- **`putAnswer` 在代码中没有被任何地方调用**，L1 的 `qa_answer` 缓存始终为空
- 正常流程未在调 LLM 之前做 L1 缓存穿透检查，无法减少 LLM 压力
