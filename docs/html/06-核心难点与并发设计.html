核心技术难点与并发设计

企业级 RAG 知识库的 8 大难点及应对策略

难点 1：文档解析精度

问题：PDF 复杂排版（多栏、表格、图片穿插）解析后结构丢失；PPT 的文字层级关系难以还原；扫描件需要 OCR。
策略：① Apache Tika 做格式统一解析 → ② PDFBox 处理复杂 PDF → ③ Tesseract 做 OCR 降级 → ④ 按文档类型注册不同的解析策略链（策略模式）

难点 2：文档分割策略

问题：切太碎丢失上下文，切太大检索不精确。不同文档类型需要不同策略。
策略：父子文档模式（Parent-Child Chunk）。Child（512 tokens）做检索，检索命中后返回 Parent（2048 tokens）作为 LLM 上下文。Markdown 按标题层级分段，PDF 按页+段落分段。
LangChain4j：DocumentSplitter + 自定义 SegmentBuilder

难点 3：混合检索 + Rerank

问题：仅向量检索 → 专业术语匹配差；仅关键词检索 → 语义理解差。需要融合两路结果。
策略：① 查询改写（LLM 扩展同义词/分解复杂问题）→ ② 稠密+稀疏双路各召回 Top-20 → ③ RRF（倒数排名融合）合并为 Top-30 → ④ BGE-Reranker 精排取 Top-5 → ⑤ 送入 LLM。
LangChain4j：EmbeddingStoreContentRetriever + WebSearchContentRetriever(ES) + 自定义 FusionRetriever

难点 4：引用溯源（Citation）

问题：LLM 生成答案后，用户需要知道"这句话来自哪个文档的第几页"。企业场景下，不可追溯的答案是没用的。
策略：① 每个 Chunk 存储来源元数据（文档 ID、页码、段落号）→ ② 检索时带出元数据 → ③ System Prompt 要求 LLM 在答案中标注引用编号 → ④ 返回时附带引用文档链接

难点 5：Embedding 模型选型

问题：中英文混合场景下，通用模型中文语义理解差；API 调用延迟不可控。
推荐方案：

BGE-M3 (BAAI)中英双语，1024维，支持稠密+稀疏双表示，本地部署
GTE-Qwen2-7B-Instruct阿里出品，中文最强之一，需 GPU
text-embedding-3-large (OpenAI)API 调用，质量高但延迟+费用，备选

难点 6：多租户数据隔离

问题：部门 A 的文件不能让部门 B 检索到（除非共享）。每个检索请求必须带租户上下文。
策略：① PostgreSQL 表 level：tenant_id + dept_id 列 → ② Milvus：Partition Key = knowledgeBaseId → ③ ES：基于 index 隔离或 alias 路由 → ④ 检索时注入 ThreadLocal 租户上下文 → ⑤ 所有查询自动过滤当前用户有权访问的知识库

难点 7：LLM 幻觉与安全

问题：知识库里没有的答案，LLM 可能会编造（幻觉）；用户可能尝试 Prompt 注入攻击。
策略：① System Prompt 约束"只根据提供的文档回答，不知道就说不知道"→ ② 答案置信度评分（检索相关性低于阈值时拒绝回答）→ ③ 输入过滤（敏感词/注入检测）→ ④ 答案二次校验（LLM 自检或规则校验）

难点 8：知识库持续更新

问题：文档更新/删除后，向量库和 ES 索引中的数据必须同步更新，否则返回过期答案。
策略：① 文档 → Chunk 时记录映射关系（doc_id → chunk_id 列表）→ ② 更新时先删旧 Chunk 再写新 Chunk → ③ 文件 MD5 指纹追踪变更 → ④ Wiki 通过 Webhook 触发增量更新 → ⑤ 定时全量对账任务

并发架构策略

WebFlux 非阻塞

对话请求全链路异步（Controller → Service → LLM API），不阻塞线程池。150 QPS 单实例轻松承载。

Spring WebFlux + Reactor

连接池配置

Milvus/ES/Redis 全用连接池，LLM API 也配置 HTTP 连接池。避免频繁建连/断连。

gRPC 连接池 + 连接复用

多级缓存

L1: Caffeine 本地缓存（热点问题）→ L2: Redis（查询缓存/会话）→ 穿透保护（布隆过滤器+空值缓存）

Caffeine + Redis + 缓存穿透保护

限流与降级

Resilience4j Rate Limiter 按用户/部门限流。LLM 超时降级为"请稍后重试"。

Resilience4j + Circuit Breaker

结论

3000 人内部知识库的并发压力远低于 C 端产品（日均 150 QPS 以内）。一个配置合理的 Spring Boot 单体 + 连接池 + Redis 缓存完全够用。

真正的瓶颈不在 Web 层，而在 LLM API 调用延迟（通常 1-5 秒）和 文档解析吞吐（大 PDF 耗时）。这两个通过异步化解决即可。
