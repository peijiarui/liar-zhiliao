存储层设计详解

会话记忆 · 向量数据 · 业务数据 · 文档文件 · 五种存储各司其职

一、会话记忆存储 — Redis Stack

为什么选 Redis？

  • ChatMemory 每次对话都要读写，读写延迟必须 < 1ms
  • 记忆窗口天然是 LRU 淘汰模式，Redis TTL + 内存淘汰完美匹配
  • Redis Stack 内置 JSON 存储，ChatMessage 序列化/反序列化零成本
  • 支持 Redis Sentinel / Cluster 实现高可用

备选方案及对比

  • PostgreSQL — 磁盘 IO 太高，会话聊天不适合频繁写表
  • 本地内存 — 服务重启丢失、无法水平扩展（你别用）
  • MongoDB — 可以但 Redis 更快，且 Redis 还能兼顾缓存/限流

LangChain4j 集成方式

// 只需实现 ChatMemoryStore 接口，注入 Redis
@Bean
public ChatMemoryStore chatMemoryStore(RedisTemplate redis) {
    return new RedisChatMemoryStore(redis);  // 替代你当前的 CustomChatMemoryStore
}
// LangChain4j 自动处理序列化，你现有代码改动 < 10 行

二、向量存储 — Milvus Standalone

部署模式Milvus Standalone
Embedding 维度1024-1536（取决于模型）
索引类型IVF_FLAT → HNSW（生产）
相似度度量Cosine Similarity
分区策略按 knowledgeBaseId 分区（多租户）

为什么不用 pgvector？

pgvector 适合小规模（<100 万向量）且不想引入额外组件的场景。

但 3000 人企业级知识库，文档可能达百万级，pgvector 索引性能会明显下降。

Milvus 专为向量检索设计，10 亿级规模、混合检索、GPU 加速。
建议：pgvector 用于开发/测试环境，生产用 Milvus。

三、Elasticsearch — 关键词检索

与 Milvus 形成混合检索（Hybrid Search）。BM25 算法对专业术语、编号、日期等精确匹配远优于向量检索。

倒排索引 全文检索 高亮片段 分面搜索

四、PostgreSQL — 业务数据主库

核心表users, departments, knowledge_bases, documents, chunks, conversations, audit_logs
多租户所有表带 tenant_id / dept_id，基于 Row-Level Security 或应用层过滤
扩展pg_trgm（模糊搜索）、pgvector（开发环境向量检索）

五、MinIO — 原始文档对象存储

S3 兼容 API，存放上传的原始文档（PDF/Word/图片等）。文档处理后，原始文件保留以供溯源和重新处理。 公司内部可部署 MinIO Server 替代 AWS S3，成本为零。

六、RabbitMQ — 异步任务

大文档（>50MB 的 PDF）解析耗时，通过消息队列异步处理，避免阻塞 HTTP 请求。也用于批量导入任务的解耦。
