技术选型 Q&A

9 个关键技术决策的深度对比与解答

Q1：会话记忆用 Redis 会丢失吗？为什么不用 MySQL？

Redis 方案（推荐）

  • RDB 快照 + AOF 日志双重持久化，不会丢数据
  • 配置：save 60 1000（60秒内1000次写入触发快照）+ appendonly yes
  • 极端情况（机房断电）最多丢失最后 1 秒的 AOF 数据
  • Redis Sentinel / Cluster 实现高可用自动故障转移

MySQL 的问题

  • 每次对话都写表 → IO 密集，磁盘延迟 5-10ms
  • 对话消失被海量读写（用户每说一句就触发 updateMessages）
  • 20 条消息窗口 = 频繁 UPDATE，MySQL 行锁竞争
  • ChatMemory 不是"持久数据"——它是短期上下文窗口，丢了不影响业务，Redis 的定位天然匹配

结论：ChatMemory 的读写模式是"高频读写 + 短期窗口 + 允许丢失"，这恰好是 Redis 的强项。如果实在不放心，可以 Redis + MySQL 双写（Redis 读，MySQL 冷备），但 3000 人企业场景下完全没必要。

Q2：Redis Stack 也支持向量检索，为什么还要 Milvus？

对比维度 Redis Stack (向量) Milvus Standalone
定位 缓存 + 轻量向量检索 专业向量数据库
索引算法 FLAT / HNSW HNSW / IVF_FLAT / IVF_PQ / SCANN / DiskANN 等 11 种
向量规模 < 100 万（受内存限制） 10 亿+（内存+磁盘混合）
混合检索 支持但能力有限 原生混合检索（稠密+稀疏+标量过滤）
分区/多租户 需手动管理 key 前缀 原生 Partition Key，自动路由
召回精度 一般 高（支持 IVF_PQ 等压缩索引仍保持高召回）
内存成本 极高！全内存存储，100万条1024维 ≈ 4GB+ 内存 低！内存+磁盘混合，热数据在内存，冷数据在磁盘
适用场景 小规模缓存、实时推荐 企业级知识库、语义搜索

结论：Redis 的向量是"锦上添花"的附加功能，不是立身之本。知识库场景下，内存成本是关键——100 万文档 Chunk 占用 4GB+ 内存放 Redis，而 Milvus 仅需几百 MB。同时 Milvus 的索引算法、多租户分区、混合检索能力都碾压 Redis Stack。

Q3：BGE-M3 本地部署是向量模型吗？为什么不自调第三方 API？

是的，BGE-M3 是 Embedding 模型（文本→向量）。它由北京智源研究院 (BAAI) 开源，HuggingFace 上可直接下载。

本地部署的优点

  • 数据不出内网——企业文档不经过第三方 API
  • 零调用成本——无 API 按量计费
  • 低延迟——内网调用 < 50ms，第三方 API 50-500ms
  • BGE-M3 特点：中英双语、1024维、同时输出稠密+稀疏向量（天然支持混合检索）

第三方 Embedding API 推荐

  • 通义千问 text-embedding-v3 — 中文效果最好，1024/768维可选
  • OpenAI text-embedding-3-large — 中英文通用，3072维
  • DeepSeek Embedding — 性价比高，适合已有 DeepSeek 用户
  • Cohere Embed v3 — 多语言，企业级 SLA

建议：MVP 阶段可先用 DeepSeek Embedding API（你已有 API Key），快速跑通。生产环境迁移到 BGE-M3 本地部署。两者 LangChain4j 切换只需改 Bean 配置，零业务代码变更。

Q4：业务数据库为什么用 PostgreSQL 而不是 MySQL？

特性 PostgreSQL 16 MySQL 8.0
JSON 支持 JSONB — 二进制存储，支持索引，查询极快 JSON — 文本存储，性能差一个数量级
全文搜索 内置 tsvector + 中文分词（zhparser） ngram 分词，中文效果差
向量扩展 pgvector — 直接支持向量检索 无原生支持
并发模型 MVCC — 读写互不阻塞 MVCC + Undo Log，高并发下略逊
分析能力 窗口函数、CTE、递归查询 窗口函数支持较晚
许可协议 完全开源 (PostgreSQL License) Oracle 所有，存在合规风险

核心原因：pgvector。知识库的文档元数据、Chunk 内容本身就存在 PG 里，pgvector 可以直接在同一张表上做向量检索，开发环境甚至不需要单独部署 Milvus。一条 SQL 搞定"查询标题包含'请假'的文档 AND 内容向量相似度 > 0.8"。MySQL 做不到。

Q5：消息队列做什么？为什么 RabbitMQ 而不是 RocketMQ？

消息队列在企业知识库中的三大用途：

  1. 文档异步处理——上传 50MB PDF 后立即返回"处理中"，Worker 异步解析+向量化+写入索引，处理完通知用户
  2. 批量导入——从 Wiki 同步 1000 篇文档时，逐条投递到队列，多 Worker 并行消费，控制速率避免打爆 LLM API
  3. 事件通知——文档索引完成、知识库更新等事件，通知下游服务（如 WebSocket 推送前端刷新）

选 RabbitMQ 的理由

  • Spring 原生集成——spring-amqp 开箱即用，配置量极低
  • 功能刚好够——ACK 确认、死信队列、延迟消息、优先级队列
  • 运维简单——Docker 一键部署，管理界面直观
  • 社区成熟——文档/案例丰富，问题好排查

RocketMQ 更合适的情况

  • 十万级 TPS 的高吞吐场景（如电商交易）
  • 需要事务消息（分布式事务）
  • 团队对阿里技术栈更熟悉
  • 已在用阿里云，RocketMQ 托管服务更省心

结论：3000 人内部知识库，日均处理文档不过百篇，RabbitMQ 的吞吐完全够用。Spring 集成度和运维简单性是决定性因素。如果公司已经在用 RocketMQ，优先复用现有基础设施。

Q6：MinIO 怎么使用？（含伪代码）

MinIO 是自建的对象存储服务，兼容 AWS S3 协议。通俗理解：你公司内部的"私有 AWS S3"。Docker 一行命令启动。Spring Boot 通过 S3 SDK 操作文件。 

// ===== 1. Docker 启动 MinIO =====
// docker run -p 9000:9000 -p 9001:9001 \
//   -e MINIO_ROOT_USER=admin -e MINIO_ROOT_PASSWORD=admin123 \
//   minio/minio server /data --console-address ":9001"

// ===== 2. Maven 依赖 =====
// io.minio:minio:8.5.7

// ===== 3. 上传文档 =====
@Service
public class DocumentStorageService {

    @Autowired
    private MinioClient minioClient;  // Bean: MinioClient.builder()
                                          //   .endpoint("http://192.168.1.100:9000")
                                          //   .credentials("admin","admin123").build()

    // 上传文档
    public String uploadDocument(MultipartFile file, String deptId) {
        String objectKey = deptId + "/" + UUID.randomUUID()
                         + "/" + file.getOriginalFilename();

        minioClient.putObject(PutObjectArgs.builder()
            .bucket("knowledge-base")      // bucket = 命名空间
            .object(objectKey)              // deptId/uuid/filename.pdf
            .stream(file.getInputStream(), file.getSize(), -1)
            .contentType(file.getContentType())
            .build());

        return objectKey;  // 存到 PG 的 documents 表
    }

    // 下载文档（用于溯源：用户点击引用链接时返回原始文件）
    public InputStream downloadDocument(String objectKey) {
        return minioClient.getObject(GetObjectArgs.builder()
            .bucket("knowledge-base")
            .object(objectKey)
            .build());
    }

    // 生成预签名 URL（前端直接下载，不经过后端）
    public String getPresignedUrl(String objectKey) {
        return minioClient.getPresignedObjectUrl(GetPresignedObjectUrlArgs.builder()
            .bucket("knowledge-base")
            .object(objectKey)
            .method(Method.GET)
            .expiry(60)  // 60秒过期
            .build());
    }
}

流程：用户上传文档 → 先存 MinIO 拿到 objectKey → 再异步解析/向量化 → objectKey 存入 PG 的 documents 表 → 用户点击"查看原文"时生成预签名 URL 返回前端。

Q7：Apache Tika + PDFBox + Tesseract 怎么用？开源吗？

全部开源（Apache 2.0 License），可以放心商用。

Apache Tika

通用文档解析引擎，自动检测文件类型并提取文本。支持 PDF/Word/PPT/Excel/HTML 等 1000+ 格式。

门面层，统一入口

PDFBox

专业 PDF 处理库。当 Tika 对复杂 PDF（多栏、加密、表单）解析不佳时，用 PDFBox 做降级处理。

PDF 深度处理

Tesseract

Google 开源的 OCR 引擎。当 PDF 是扫描件/图片时，无法直接提取文字，需要 Tesseract 光学识别。

OCR 文字识别 
// ===== Maven =====
// org.apache.tika:tika-core:2.9.2
// org.apache.tika:tika-parsers-standard-package:2.9.2
// org.apache.pdfbox:pdfbox:3.0.1
// net.sourceforge.tess4j:tess4j:5.9.0  (Tesseract Java 封装)

// ===== 1. Tika 通用解析（覆盖 80% 场景）=====
public String parseWithTika(InputStream fileStream, String fileName) {
    Tika tika = new Tika();
    return tika.parseToString(fileStream);  // 自动检测类型，提取纯文本
}

// ===== 2. PDFBox 深度处理（复杂 PDF）=====
public String parsePdfWithPDFBox(InputStream fileStream) {
    PDDocument document = PDDocument.load(fileStream);
    PDFTextStripper stripper = new PDFTextStripper();
    stripper.setSortByPosition(true);  // 按页面位置排序，保留阅读顺序
    return stripper.getText(document);
}

// ===== 3. Tesseract OCR（扫描件降级）=====
public String parseScannedPdfWithOCR(InputStream fileStream) {
    PDDocument document = PDDocument.load(fileStream);
    PDFRenderer renderer = new PDFRenderer(document);
    StringBuilder text = new StringBuilder();
    for (int page = 0; page < document.getNumberOfPages(); page++) {
        BufferedImage image = renderer.renderImageWithDPI(page, 300);  // 300 DPI
        text.append(tesseract.doOCR(image));  // Tesseract OCR
    }
    return text.toString();
}

// ===== 4. 策略链（实际使用方式）=====
public String parseDocument(DocumentFile doc) {
    String text = parseWithTika(doc.getStream(), doc.getFileName());

    // 如果 Tika 解析结果为空或太少（说明是扫描件），走 OCR
    if (text == null || text.trim().length() < 50) {
        if (doc.getType() == FileType.PDF) {
            text = parseScannedPdfWithOCR(doc.getStream());
        }
    }
    return text;
}

策略模式设计：为每种文件类型注册不同的解析策略（WordHandler / PdfHandler / ExcelHandler / MarkdownHandler），Tika 先尝试通用解析，失败则走类型专属的降级策略。全部开源免费。

Q8：Caffeine + Redis 多级缓存用在哪？怎么实现？

三处核心用途

  1. 热点问题缓存——"怎么请假"这种高频问题，缓存答案避免重复调用 LLM（成本最高的环节）
  2. Embedding 缓存——相同文本的向量化结果缓存，避免重复调用 Embedding 模型
  3. 权限/用户信息——SSO 登录后的用户信息、部门权限，缓存减少 DB 查询

两级缓存架构

L1: Caffeine（本地 JVM 内存）
→ 容量 1000 条，TTL 5分钟，极速 < 0.01ms

L2: Redis（分布式共享）
→ 容量 10 万条，TTL 1小时，跨实例共享 
// ===== 多级缓存实现（基于 Spring Cache + Caffeine + Redis）=====

@Service
public class QuestionCacheService {

    @Autowired private CacheManager caffeineCache;   // L1: Caffeine
    @Autowired private RedisTemplate<String, String> redis;  // L2: Redis
    @Autowired private ChatService chatService;          // LLM 调用

    // 多级缓存查询
    public String getAnswer(String question, String deptId) {
        String cacheKey = "qa:" + deptId + ":" + hash(question);

        // L1: 查 Caffeine（0.01ms）
        Cache caffeine = caffeineCache.getCache("questions");
        String cached = caffeine.get(cacheKey, String.class);
        if (cached != null) return cached;

        // L2: 查 Redis（1ms）
        cached = redis.opsForValue().get(cacheKey);
        if (cached != null) {
            caffeine.put(cacheKey, cached);  // 回填 L1
            return cached;
        }

        // L3: 调用 LLM（1000-5000ms）
        String answer = chatService.chat(cacheKey, question)
            .collectList().block().stream().collect(Collectors.joining());

        // 同时写入 L1 和 L2
        caffeine.put(cacheKey, answer);
        redis.opsForValue().set(cacheKey, answer, Duration.ofHours(1));

        return answer;
    }

    private String hash(String question) {
        // 对问题做语义哈希（相似问法映射到同一缓存 key）
        // 简单版：MD5；高级版：先用 Embedding 找到最相似已缓存问题
        return DigestUtils.md5Hex(question.trim().toLowerCase());
    }
}

命中率预期：L1 Caffeine 命中率 30-40%（热点问题重复问），L2 Redis 命中率 60-70%（跨用户共享），总缓存命中率约 70%，即可省 70% 的 LLM API 费用。

Q9：Prometheus + Grafana + Actuator 监控什么？怎么落实？

监控什么？

LLM 调用次数、延迟、成功率、Token 消耗
检索性能召回率、延迟、Top-K 命中率
并发与错误QPS、响应时间、错误率、限流触发
基础设施JVM 内存/GC、Redis/Milvus/ES 健康
文档处理队列积压、处理成功率、处理耗时

怎么落实？

  1. Actuator + Micrometer — Spring Boot 内置，暴露 /actuator/prometheus 端点，自动采集 JVM/HTTP/DB 指标
  2. 自定义指标 — 用 Micrometer 的 @Timed、Counter、Gauge 注解埋点（LLM 调用耗时、检索次数、缓存命中率）
  3. Prometheus — 定期抓取 /actuator/prometheus，存储时序数据
  4. Grafana — 连接 Prometheus 数据源，搭建仪表盘，配置告警规则（如错误率 > 5% 发钉钉/邮件）
// ===== 1. Maven 依赖 =====
// org.springframework.boot:spring-boot-starter-actuator
// io.micrometer:micrometer-registry-prometheus

// ===== 2. application.yaml =====
// management.endpoints.web.exposure.include: health,prometheus,metrics
// management.metrics.export.prometheus.enabled: true

// ===== 3. 自定义埋点示例 =====
@Service
public class RetrievalService {

    private final MeterRegistry meterRegistry;

    // 声明计数器：检索总次数
    private final Counter retrievalCounter;

    // 声明直方图：检索耗时分布
    private final Timer retrievalTimer;

    public List<Hit> retrieve(Query query) {
        retrievalCounter.increment();  // 计数 +1

        return retrievalTimer.record(() -> {
            // 实际检索逻辑...
            return milvusClient.search(query);
        });
    }
}

// ===== 4. Grafana 仪表盘（核心面板）= =====
// - QPS 曲线 (rate(http_requests_total[1m]))
// - LLM 调用延迟 P50/P95/P99 (histogram_quantile)
// - 检索延迟 P50/P95/P99
// - 缓存命中率 = 缓存命中次数 / 总查询次数
// - 错误率 = 错误请求 / 总请求
// - JVM 堆内存使用率 & GC 频率
// - 告警规则：错误率 > 5% | P99 > 5s | 内存 > 85%

落实步骤：① 加依赖 + YAML 配置（10分钟）→ ② 核心服务埋点（30分钟）→ ③ Docker 启动 Prometheus + Grafana → ④ 导入 Spring Boot 仪表盘模板（Grafana 社区有现成的 JVM 面板 ID: 4701）→ ⑤ 配置钉钉/邮件告警。
