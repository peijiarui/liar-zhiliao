推荐技术栈：混合务实路线

3000人企业 · Spring Boot + LangChain4j · 模块化单体起步，渐进式演进微服务

为什么不选纯微服务？

过度设计的代价

  • 3000人内部系统，峰值并发约 100-300 QPS，单体完全可以承载
  • 微服务增加运维复杂度、调试难度、事务一致性成本
  • 团队初期规模有限，微服务的收益（独立部署/独立扩缩容）暂时用不上

推荐架构策略

当前阶段：模块化单体

  • Spring Boot 单应用，Maven 多模块拆分
  • 包结构隔离（controller → service → repository）
  • 数据库内部分库分 schema（租户隔离）

未来演进：按需拆分

  • 检索服务 → 独立（计算密集型）
  • 文档处理 → 独立（IO 密集型）
  • 对话服务 → 独立（高并发）
